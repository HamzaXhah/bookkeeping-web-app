# CLAUDE.md

This file gives Claude Code the context it needs to work effectively in this repo. Read it first before making any changes.

## Project

**Simple Bookkeeping & P&L Tool** — a self-use web app that turns bank/credit-card CSVs into a clean Profit & Loss report, with AI-powered transaction categorization.

This is a **personal tool**, not a product. Single user, no auth, no multi-tenancy. The architecture is intentionally minimal — fewer moving parts over flexibility, near-zero ops over scale.

Full architecture is documented in `docs/architecture.docx`. Read it if you need context beyond this file.

## Core Principles

1. **Stay minimal.** Do not add features that aren't in the proposal or architecture doc. If a request feels like scope creep, ask before building.
2. **Self-use, not production.** No auth, no rate limiting, no observability stack, no background jobs. Don't add them unless explicitly asked.
3. **Local-first, hosted-portable.** The system runs on a single laptop today (SQLite). It must remain trivial to migrate to Supabase + Vercel later by swapping only the DB driver. Don't introduce anything that breaks this portability.
4. **The URL is the source of truth for the active business.** Never store "current business" in cookies, localStorage, or React context.
5. **Server holds secrets, never the browser.** The Claude API key lives in `.env.local` and is only ever read inside Next.js Route Handlers.

## Tech Stack (locked — do not substitute)

- **Framework:** Next.js (App Router) + TypeScript
- **UI:** Tailwind CSS + shadcn/ui
- **Database:** SQLite via `better-sqlite3` (v1) — design must support `postgres` driver swap for hosted mode
- **ORM:** Drizzle ORM (with `drizzle-kit` for migrations)
- **AI:** Claude Haiku 4.5 via `@anthropic-ai/sdk` — server-side only
- **CSV parsing:** Papa Parse
- **Fuzzy matching:** `fastest-levenshtein`
- **IDs:** `cuid2`
- **Exports:** `jspdf` + `jspdf-autotable` + `xlsx` — **client-side only**
- **API style:** File-based Next.js Route Handlers (`app/api/*/route.ts`). Not Server Actions.

If a task seems to need something outside this list, ask first.

## Data Model

Four tables. All FKs cascade on delete.

- **businesses** — `id` (cuid2), `name`, `created_at`
- **categories** — `id`, `business_id` (FK), `name`, `kind` ('income' | 'expense'), unique on (business_id, name)
- **transactions** — `id`, `business_id`, `date` (ISO yyyy-mm-dd string), `description`, `amount` (signed REAL: + income, − expense), `category_id` (nullable, ON DELETE SET NULL), `dedup_hash`, `source_file`, `created_at`. Unique on (business_id, dedup_hash). Index on (business_id, date).
- **merchant_memory** — `id`, `business_id`, `normalized_desc`, `category_id`, `updated_at`. Unique on (business_id, normalized_desc).

**Key conventions:**
- All IDs are cuid2 strings.
- Dates are ISO strings, not timestamps. The tool deals in calendar dates.
- Amounts are a single signed `REAL` — sign carries direction. Don't add a separate type column.

## Key Flows

### CSV Import (`POST /api/import`)
1. Parse CSV with Papa Parse; auto-detect date/description/amount columns. Merge debit+credit columns into a signed amount if both exist.
2. Hash each row: `sha256(business_id|date|amount|description)`. Skip rows whose hash already exists for the business.
3. Memory match: normalize description (lowercase, strip punctuation and standalone digits), then exact-match on `merchant_memory`. If no exact match, run Levenshtein and accept the best match if similarity ≥ 0.85.
4. AI batch: send remaining uncategorized rows to Claude Haiku in chunks of ~50. Prompt includes the business's category list. Response is a JSON array of `{ index, category }`.
5. Insert all rows in a single Drizzle transaction.
6. Return counts: `{ imported, skipped_duplicates, ai_categorized, memory_categorized }`.

### Manual category change (`PATCH /api/transactions/[id]`)
- Update the transaction's `category_id`.
- **Upsert `merchant_memory`** keyed on `(business_id, normalized_desc)` with the new category.
- Both writes happen in one DB transaction.

### Manual re-run (`POST /api/categorize`)
- Takes a list of transaction ids.
- Runs only the AI batch step for those rows.
- **Does not write to merchant_memory** — the user requested a fresh AI pass, not a correction.

### P&L (`GET /api/pnl?business_id=X&from=...&to=...`)
- Single GROUP BY query joining transactions to categories.
- Returns `{ range, income: [...], expenses: [...], totals: { income, expenses, net } }`.

### Exports
- Browser fetches `/api/pnl`, then jsPDF or SheetJS builds the file in-memory and triggers a download.
- Server is never involved in rendering exports.
- Use dynamic `import()` for jspdf/xlsx to keep the initial page bundle small.

## AI Integration Rules

- `ANTHROPIC_API_KEY` is read from `.env.local` at server startup. Never expose it to the browser. Never log it.
- Only import `@anthropic-ai/sdk` inside files under `app/api/` or `lib/ai/`. If you import it elsewhere, you've made a mistake.
- One prompt template, used by both auto-on-import and manual re-run. Live in `lib/ai/categorize.ts`.
- Batch up to ~50 transactions per Claude call. Split larger imports into chunks. No streaming, no parallel fan-out.
- On AI failure (network, 5xx, malformed JSON): save affected rows with `category_id = null`. The import still succeeds. **No retry logic, no exponential backoff** — the manual re-run button covers this case.

## Database Rules

- Schema lives in `lib/db/schema.ts`. It is the single source of truth.
- Use `drizzle-kit generate` to create migrations. Never write migrations by hand.
- Use `drizzle-kit migrate` to apply them.
- The DB connection in `lib/db/client.ts` must be the only place that imports `better-sqlite3`. Everything else imports the Drizzle instance.
- Do not write raw SQL strings unless absolutely needed. Use Drizzle's query builder.

## Routing & State Rules

- Pages that operate on business data live under `app/business/[id]/`. Read the id from route params.
- The layout at `app/business/[id]/layout.tsx` fetches the business, renders the switcher header, and provides the business name/id to children via props or a server component.
- **Do not** add a global "current business" store (Zustand, Context, cookie, localStorage). The URL is it.
- Switching businesses is a navigation (`<Link>` or `router.push`), not a state update.

## What NOT to Build

These were explicitly cut from scope. Don't add them on your own initiative:

- Authentication, login, sessions, password reset, anything user-account-related
- Direct bank-feed integrations (Plaid, etc.) — CSV upload only
- Invoicing, estimates, payroll, tax filing
- Multi-user collaboration, sharing, comments
- Email, notifications, scheduled jobs, cron
- Rate limiting, CAPTCHA, abuse prevention
- Structured logging, metrics, tracing, error monitoring services
- File upload buckets — CSVs are read in-memory and discarded
- Background job queues — imports run synchronously in the request

If a feature request implies any of the above, ask before implementing.

## Code Style

- TypeScript strict mode. No `any` unless there's a comment explaining why.
- Server-only files: add `import "server-only";` at the top of files that must never run in the browser (DB client, AI client, anything reading env vars).
- Validate inputs at API boundaries with Zod. Don't trust request bodies.
- Keep route handlers thin: they should orchestrate, not contain business logic. Logic goes in `lib/`.
- Pure functions in `lib/pnl/aggregate.ts` and `lib/csv/parse.ts` — these are the most testable units.
- shadcn/ui components: pull them in with `npx shadcn@latest add <name>` only when needed. Don't pre-add a kitchen sink.

## Migration Path to Hosted (Reference)

When the user later asks to deploy to Supabase + Vercel, the migration is:

1. `npm uninstall better-sqlite3 @types/better-sqlite3 && npm install postgres`
2. Update `lib/db/client.ts` to use the `postgres` driver pointed at `DATABASE_URL`.
3. Run `drizzle-kit generate` against the Postgres dialect, apply once on Supabase.
4. Move `.env.local` vars (`ANTHROPIC_API_KEY`, `DATABASE_URL`) to Vercel project env vars.
5. `vercel deploy`.

That's the whole list. No code changes outside `lib/db/client.ts`. Keep the design consistent with this so it stays true.
